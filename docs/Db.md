# Сводка по архитектуре БД - TaskFlow Monitor

## 🎯 ОСНОВНАЯ КОНЦЕПЦИЯ

**Трехуровневая иерархия:** 
```
User → Team → Project → Task → Dependency → Action
```
Где каждый уровень строго отделен и имеет свои правила доступа.

---

## 🧩 МОДУЛЬ 1: ПОЛЬЗОВАТЕЛИ (user.py)

### Сущности:
1. **UserRole** - глобальные роли в системе (клиент, админ, тестер)
2. **User** - пользователи сервиса
3. **AuthSession** - сессии аутентификации
4. **RecoveryCode** - коды восстановления
5. **AuthLog** - логирование аутентификации

### Ключевые правила:
- `UserRole` не имеет отношения к проектам/командам — это глобальный статус
- Подтверждение email: 6-значный код, лимит попыток и срок жизни кода (поля `email_code`, `email_code_expires`, `email_code_attempts`), флаг `email_verified`
- Сессии: access token ~30 мин, refresh token 7 дней (см. `core/config.py`)
- У пользователя может быть много команд и много проектов

---

## 🏢 МОДУЛЬ 2: КОМАНДЫ (team.py)

### Сущности:
1. **TeamMemberRole** - роли внутри команды (owner, admin, member)
2. **Team** - команда пользователей
3. **TeamMember** - связь пользователя с командой
4. **TeamInvitation** - приглашения в команду

### Ключевые правила:

**Права ролей в команде:**
- **Owner** (priority 100): может всё (управление командой, проектами, участниками)
- **Admin** (priority 80): управление проектами, приглашение/удаление участников
- **Member** (priority 50): только работа в проектах

**Код приглашения в команду:**
- Генерируется автоматически, уникальный
- Живет 1 час
- Обновляется событийно (при запросе данных, если истек)
- Любой, у кого есть код, может вступить

**Правила вступления:**
- Только зарегистрированные пользователи
- При вступлении автоматически создается TeamMember с ролью 'member'
- Приглашение через Invitations - для назначения конкретной роли

---

## 📊 МОДУЛЬ 3: ПРОЕКТЫ (project.py)

### Сущности:
1. **ProjectRole** - роли внутри проекта (owner, manager, developer, observer)
2. **Project** - проект, принадлежащий команде
3. **ProjectMember** - участники проекта
4. **ProjectInvitation** - приглашения в проект

### Ключевые правила:

**Права ролей в проекте:**

| Роль | priority | Созд. задач | Ред. чужие | Уд. чужие | Созд. связей | Упр. участниками |
|------|----------|-------------|------------|-----------|--------------|------------------|
| **Owner** | 100 | ✅ | ✅ | ✅ | ✅ | ✅ |
| **Manager** | 80 | ✅ | ✅ | ✅ | ✅ | ✅ |
| **Developer** | 60 | ✅ | ❌ | ❌ | ✅ | ❌ |
| **Observer** | 40 | ❌ | ❌ | ❌ | ❌ | ❌ |

**ВАЖНО:** Developer может:
- Создавать свои задачи ✅
- Редактировать ТОЛЬКО свои задачи ✅
- Создавать зависимости с ЛЮБЫМИ задачами (чужими в том числе) ✅
- НЕ может удалять чужие задачи ❌
- НЕ может удалять зависимости ❌

**Наследование участников:**
- Участник должен быть членом команды-владельца проекта
- Нельзя добавить в проект человека, не состоящего в команде
- Приглашение в проект = выбор участника команды + назначение роли

---

## ✅ МОДУЛЬ 4: ЗАДАЧИ И ГРАФ (task.py)

### Сущности:
1. **TaskStatus** - статусы задач (todo, in_progress, review, completed, blocked)
2. **DependencyActionType** - типы действий на ребрах
3. **Task** - задача (составной ключ project_id + id)
4. **TaskDependency** - зависимость между задачами (ребро графа)
5. **DependencyAction** - действие на ребре (может быть несколько)
6. **TaskEvent** - лог событий задачи
7. **ScheduledAction** - отложенные действия

### Ключевые правила:

**Составной ключ Task:**
```python
PRIMARY KEY (project_id, id)  # id автоинкремент в рамках таблицы
```
- Номер задачи уникален в пределах проекта
- Во всех связях всегда передается пара (project_id, task_id)

**Хранение графа:**
- **НЕ бинарный формат** ❌
- **Два представления:**
  1. `Project.graph_data` - полный JSON экспорт из ReactFlow (для рендеринга)
  2. `TaskDependency` - реляционная таблица связей (для бизнес-логики)

**Действия на ребрах:**
- Одно ребро = МНОГО действий
- Типы действий:
  - `notify_assignee` - уведомить исполнителя целевой задачи
  - `notify_creator` - уведомить создателя исходной задачи  
  - `notify_custom` - уведомить конкретного пользователя
  - `change_status` - изменить статус задачи
  - `create_subtask` - создать подзадачу
- Поддержка отложенного выполнения (delay_minutes)

**Обработка событий:**
- **НЕТ бесконечного воркера**, который сканирует граф ❌
- **ТОЛЬКО реакция на действия пользователя** ✅
- При изменении статуса задачи → проверяем исходящие ребра → выполняем действия
- При завершении задачи → проверяем входящие ребра → обновляем readiness

**Дедлайны и отложенные уведомления:**
- ЕДИНСТВЕННЫЙ воркер - раз в минуту проверяет `ScheduledAction`
- Планируются: уведомления за 24/1 час до дедлайна, просрочка
- Всё остальное - синхронно

**Readiness задачи:**
- Задача "готова к выполнению", когда ВСЕ её предшественники выполнены
- Вычисляется динамически при проверке
- Не хранится в БД как поле (только кэш)

## 🔐 СКВОЗНЫЕ ПРАВИЛА ДОСТУПА

### Уровень 1: Команда
```
- Вступить по коду: может любой аутентифицированный
- Пригласить: Admin, Owner
- Удалить участника: Admin, Owner
- Создать проект: Admin, Owner
```

### Уровень 2: Проект
```
- Создать задачу: все, кроме Observer
- Редактировать задачу: 
  • Owner/Manager: любые
  • Developer: только свои (creator_id == me или assignee_id == me)
- Удалить задачу: только Owner/Manager
- Создать зависимость:
  • Owner/Manager: любые
  • Developer: если source_task - мой ИЛИ target_task - мой
- Удалить зависимость: только Owner/Manager
- Управлять участниками: только Owner/Manager
```

### Уровень 3: Задача
```
- Изменить статус: 
  • Исполнитель: может менять
  • Создатель: может менять
  • Manager/Owner: могут менять
- Назначить исполнителя: 
  • Создатель задачи
  • Manager/Owner
```

## ⚡ ПАТТЕРНЫ РАБОТЫ С ГРАФОМ

### 1. Чтение графа для отображения:
```sql
-- Полный граф для ReactFlow
SELECT graph_data FROM projects WHERE id = ?

-- Или сборка на лету (если нужно актуальное состояние статусов)
SELECT * FROM tasks WHERE project_id = ?
SELECT * FROM task_dependencies WHERE project_id = ?
```

### 2. Проверка готовности задачи:
```sql
-- Все предшественники выполнены?
SELECT COUNT(*) FROM task_dependencies d
LEFT JOIN tasks t ON t.id = d.source_task_id
WHERE d.target_task_id = ? AND d.project_id = ?
  AND (t.status_id != (SELECT id FROM task_statuses WHERE name = 'completed'))
```

### 3. Выполнение действий при завершении:
```python
1. Найти все исходящие ребра от completed task
2. Для каждого ребра получить все DependencyAction
3. Выполнить действия (уведомления, смены статусов)
4. Для каждой целевой задачи проверить readiness
5. Если стала ready - опционально уведомить исполнителя
```

### 4. Отложенные действия:
```python
1. При создании задачи с дедлайном → создать ScheduledAction
2. Раз в минуту: SELECT * FROM scheduled_actions 
   WHERE scheduled_for <= NOW() AND status = 'pending'
3. Выполнить, обновить статус
```

## 🚫 ЧЕГО НЕ ДЕЛАТЬ (ЗАПОМНИТЬ!)

1. **НЕ** хранить граф в бинарном виде ❌
2. **НЕ** запускать бесконечный цикл сканирования графа ❌
3. **НЕ** хранить readiness задачи в поле (только вычислять или кэшировать) ❌
4. **НЕ** давать Developer права на удаление чужих задач ❌
5. **НЕ** добавлять в проект человека не из команды ❌
6. **НЕ** делать составной ключ без автоинкремента (id сам растет) ❌
7. **НЕ** использовать нейросети для обхода графа (оверинжиниринг) ❌

---

## ✅ ЧТО ДЕЛАТЬ (ПАТТЕРНЫ)

1. **Реагировать на события** - только при действиях пользователя
2. **Кэшировать частые запросы** - Redis или in-memory для графа
3. **Логировать всё** - TaskEvent для аудита
4. **Проверять права** - на каждом уровне отдельно
5. **Использовать JSON** - для гибких настроек и метаданных
6. **Денормализовать счетчики** - members_count, tasks_count для быстрых списков
7. **Индексировать поиск** - все поля WHERE и JOIN должны быть в индексах

---

Эта архитектура дает **полное разделение ответственности**, **гибкую систему прав** и **максимальную производительность** без овер инжиниринга.
