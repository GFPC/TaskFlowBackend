# 🎯 ОБЩИЕ ПРИНЦИПЫ

## 1.1 Базовые правила взаимодействия

| Ситуация | Стратегия | Почему |
|----------|-----------|--------|
| **Часто запрашиваемые, редко изменяемые** | Кэшировать + фоновая инвалидация | Профиль, настройки, список команд |
| **Динамические, часто изменяемые** | Запрашивать при каждом рендере | Статусы задач, граф зависимостей |
| **Списочные данные с пагинацией** | Запрашивать пачками (limit/offset) | Список задач, участников |
| **Единичные операции** | Отдельные запросы | Создание/удаление/обновление |
| **Bulk-операции** | Один запрос на пакет | Массовое изменение статусов |

## 1.2 HTTP-методы и их семантика

```
GET     - Получение данных (идемпотентно, безопасно)
POST    - Создание ресурса (не идемпотентно)
PUT     - Полная замена ресурса (идемпотентно)
PATCH   - Частичное обновление (идемпотентно)
DELETE  - Удаление ресурса (идемпотентно)
```

## 1.3 Формат ответов

**Успех (2xx):**
```json
{
  "id": 123,
  "name": "...",
  // данные ресурса
}
```

**Ошибка (4xx, 5xx):**
```json
{
  "detail": "Human-readable error message"
}
```

# 🔐 АУТЕНТИФИКАЦИЯ И ПОЛЬЗОВАТЕЛИ

## 2.1 Процесс регистрации и входа (email OTP)

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Регистрация│────▶│ Код на email│────▶│verify-email │────▶│  Сессия +   │
│  (обязат.   │     │  6 цифр     │     │             │     │   токены    │
│   email)    │     │             │     │             │     │             │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

### Критичные моменты

1. **После регистрации пользователь не получает токены сразу** — только `user_id`, флаг `email_sent` и при необходимости `verification_code` (см. ниже).
2. **Подтверждение:** `POST /api/v1/auth/verify-email` с телом `{ "user_id", "code" }` (6 цифр из письма).
3. **При логине**, если email ещё не подтверждён на бэкенде, ответ аналогичен: `requires_verification: true`, новый код уходит на почту, в ответе могут быть `email_sent`, опционально `verification_code` (если включён DEBUG на сервере или письмо не отправилось).
4. **Не полагайтесь на код в JSON в продакшене** — в обычном режиме код только в письме. UI: форма ввода 6 цифр + текст «Проверьте почту (и спам)».
5. **Обновление access-токена** — до истечения срока (см. `expires_at` в ответах), не дожидаясь 401.

## 2.2 Профиль и настройки

### ⚡ ОПТИМИЗАЦИЯ:

✅ **ДЕЛАТЬ:**
- Кэшировать профиль при загрузке приложения
- Обновлять только после явных действий
- Использовать Optimistic Updates при изменении

❌ **НЕ ДЕЛАТЬ:**
- Запрашивать профиль при каждом переходе
- Держать открытыми несколько сессий в одном браузере

### 📊 Стратегия запросов:

| Данные | Частота | Стратегия |
|--------|---------|-----------|
| Профиль | 1 раз при старте | Кэш + фоновая проверка |
| Настройки темы | 1 раз при старте | Кэш навсегда |
| Настройки уведомлений | При изменении | Optimistic update |
| Список сессий | При входе, при открытии настроек | Отдельный запрос |

# 🔌 ИНТЕГРАЦИЯ: email, граф, восстановление пароля

Практические шаги для связки UI с API: что вызывать, в каком порядке, какие поля обрабатывать.

## Почта и OTP (регистрация и вход)

| Шаг | Метод | URL | Тело |
|-----|--------|-----|------|
| Регистрация | POST | `/api/v1/auth/register` | `first_name`, `last_name`, `username`, `password`, `email` (обязательный) |
| Подтверждение email | POST | `/api/v1/auth/verify-email` | `user_id`, `code` (6 символов) |
| Вход | POST | `/api/v1/auth/login` | `username`, `password` |
| Обновление токена | POST | `/api/v1/auth/refresh` | `refresh_token` |

**Ответ регистрации** (`EmailCodeResponse`): `user_id`, `requires_verification: true`, `email_sent`, опционально `verification_code` — последнее только если на сервере `DEBUG` или письмо не отправлено (тогда покажите код пользователю вручную).

**Ответ входа при неподтверждённом email** (`LoginResponse`): `requires_verification: true`, `user_id`, опционально `verification_code`, `email_sent`.

**Профиль** в `GET /api/v1/users/me`: используйте `email_verified` (не `tg_verified`).

## Восстановление пароля

| Шаг | Метод | URL | Примечание |
|-----|--------|-----|------------|
| Запрос кода | POST | `/api/v1/auth/recovery/initiate` | Тело: `{ "username": "..." }` (поле `username` в JSON) |
| Новый пароль | POST | `/api/v1/auth/recovery/reset` | `recovery_code`, `new_password` |

Ответ `initiate`: при `success: true` — `email_sent`, `expires_at`, опционально `recovery_code` (как с OTP: в продакшене обычно не отдаётся, если письмо ушло). Текст для пользователя: код пришёл на **email**, привязанный к аккаунту.

## Граф задач (React Flow)

Базовый префикс: `/api/v1/projects/{project_slug}/tasks`

| Действие | Метод | Путь | Тело / ответ |
|----------|--------|------|----------------|
| Загрузить граф | GET | `.../tasks/graph` | `{ nodes, edges, viewport }` — `nodes[].id` строки, в `data` задачи есть числовой `id` |
| Сохранить граф | PUT | `.../tasks/graph` | То же JSON: узлы, рёбра, viewport (debounce 300–500 ms при перетаскивании) |
| Создать зависимость | POST | `.../tasks/{task_id}/dependencies` | `target_task_id`, при необходимости `dependency_type`, `description` |
| Удалить ребро | DELETE | `.../tasks/dependencies/{dependency_id}` | — |

**Ребро в ответе графа** (упрощённо): `id`, `source`, `target`, `type` (тип зависимости), `data` может содержать `description`, `animated` для визуала. Отдельных REST-эндпоинтов для «действий на ребре» в текущем API нет — бизнес-логика зависимостей на стороне задач и статусов.

**Рекомендации UI:** после `PUT /graph` не перезагружайте весь граф без необходимости; после создания/удаления зависимости можно точечно обновить данные или снова вызвать `GET .../graph` с кэшем на 15–30 сек.

## Переменные окружения бэкенда (для DevOps)

См. `core/config.py`: `RESEND_API_KEY` **или** SMTP (`SMTP_HOST`, `smtp.timeweb.ru` и порт `2525` при типичной конфигурации Timeweb), `EMAIL_FROM`, `EMAIL_PASSWORD` / `SMTP_PASSWORD`, опционально `SMTP_USER`.

# 👥 КОМАНДЫ (TEAMS)

## 3.1 Иерархия данных

```
Team
├── Members (участники)
│   ├── Owner (1)
│   ├── Admins (0+)
│   └── Members (0+)
├── Projects (0+)
├── Invitations (приглашения)
└── Invite Code (публичный код)
```

## 3.2 Роли и права - UI/UX РЕКОМЕНДАЦИИ

### 👑 Владелец (Owner)
```
✅ ВИДИТ: Всё
✅ МОЖЕТ: Управлять командой, проектами, участниками
✅ ПОКАЗЫВАТЬ: Все кнопки управления
✅ РАЗРЕШАТЬ: Удаление команды, передачу владения
```

### 🛡️ Администратор (Admin)
```
✅ ВИДИТ: Всё
✅ МОЖЕТ: Управлять проектами, участниками
❌ НЕ МОЖЕТ: Управлять командой, удалять команду
✅ ПОКАЗЫВАТЬ: Управление проектами, участниками
❌ СКРЫВАТЬ: Настройки команды, удаление
```

### 👷 Участник (Member)
```
✅ ВИДИТ: Все проекты команды
✅ МОЖЕТ: Работать в проектах
❌ НЕ МОЖЕТ: Управлять командой, добавлять участников
❌ СКРЫВАТЬ: Все кнопки управления командой
```

## 3.3 Оптимизация запросов

### ✅ ДЕЛАТЬ:

**При загрузке списка команд:**
```javascript
// ОДИН запрос на все команды пользователя
GET /api/v1/teams
// НЕ ДЕЛАТЬ отдельный запрос для каждой команды!
```

**При открытии команды:**
```javascript
// ОДИН запрос на детальную информацию
GET /api/v1/teams/{slug}
// В ответе уже ВСЕ участники и права!
// НЕ ДЕЛАТЬ отдельно /members
```

### ❌ НЕ ДЕЛАТЬ:

```javascript
// ПЛОХО - N+1 запросов
const teams = await getTeams();
for (const team of teams) {
  await getTeamMembers(team.slug); // Отдельный запрос для каждой!
}
```

### 📊 Стратегия для кода приглашения:

```javascript
// ✅ ПРАВИЛЬНО
// Получаем код (автообновляется при истечении)
const code = await getInviteCode(teamSlug);
// Показываем пользователю
// Код живет 1 час

// ✅ ПРИ ВСТУПЛЕНИИ
// Один запрос - вступление и получение членства
const result = await joinByCode(code);
// Сразу получаем team и member
```

## 3.4 Приглашения - UX стратегия

```
┌──────────────┐
│ Приглашение  │
│  по коду     │────▶ Копировать/Поделиться
│  (публичное) │      Живет 1 час
└──────────────┘
        │
        ▼
┌──────────────┐
│ Приглашение  │
│  по username │────▶ Персональное, 7 дней
│  (личное)    │      Можно отменить
└──────────────┘
```

### 🎯 Рекомендации:

1. **Публичный код:**
   - Показывать всегда, обновлять по кнопке
   - Добавить таймер обратного отсчета
   - Автоматически обновлять при истечении

2. **Персональные приглашения:**
   - Создавать через поиск пользователя
   - Отображать статус (pending/accepted/declined)
   - Давать возможность отменить

3. **Массовые приглашения:**
   - Лучше создать код и разослать ссылку
   - Избегать 10+ персональных приглашений

# ✅ ЗАДАЧИ И ГРАФ (TASKS & GRAPH)

### 📊 Структура графа от бэкенда:

```javascript
{

    {
      "id": "123",           // Строка! Всегда строка
      "type": "taskNode",    // Кастомный тип
      "data": {
        "id": 123,          // Числовой ID для запросов
        "name": "Задача",
        "status": "in_progress",
        "status_color": "#1976d2",
        "assignee": "username",
        "creator": "creator",
        "priority": 1,
        "deadline": "2026-03-01",
        "is_ready": false   // ВАЖНО! Флаг готовности

    {
      "id": "e123-456",

      "type": "blocks",
      "data": {
        "description": "опционально, текст из БД"
      },
      "animated": true,

```

## 5.2 КРИТИЧЕСКИ ВАЖНО - Оптимизация графа

### 🚨 ПРОБЛЕМА: Граф может быть огромным (1000+ узлов)

### ✅ РЕШЕНИЕ - ЛЕНИВАЯ ЗАГРУЗКА:

```javascript
// 1. Загружаем ТОЛЬКО видимую область
// Бэкенд отдает все узлы, но ReactFlow рендерит только видимые

// 2. НЕ ДЕЛАТЬ запрос при каждом панорамировании!
// Только при изменении данных

// 3. Кэшировать граф в памяти
let graphCache = {
  timestamp: Date.now(),
  data: graphData
};

// Инвалидация через 30 секунд
if (Date.now() - graphCache.timestamp > 30000) {
  await loadGraph();
}
```

### 📊 СТРАТЕГИЯ ЗАПРОСОВ ГРАФА:

| Сценарий | Действие | Оптимизация |
|----------|----------|-------------|
| Загрузка проекта | `GET /api/v1/projects/{slug}/tasks/graph` | Кэш 15–30 сек |
| Сохранение позиций | `PUT` тот же путь | Debounce 300–500 ms |
| Изменение статуса | `POST .../tasks/{id}/status` | Optimistic update |
| Создание задачи | `POST .../tasks` | Добавить узел локально |
| Удаление задачи | `DELETE .../tasks/{id}` | Убрать узел и связанные рёбра локально |
| Создание связи | `POST .../tasks/{id}/dependencies` | Добавить ребро локально |

## 5.3 Статусы задач и их визуализация

```
🎯 todo       - Серый, пунктирная рамка
⚙️ in_progress - Синий, анимированная иконка
🔍 review     - Оранжевый, иконка проверки
✅ completed  - Зеленый, зачеркнутый текст
🚫 blocked    - Красный, иконка блокировки
```

### 🎯 Правила отображения:

1. **is_ready = true** (задача готова к выполнению):
   - Подсветка зеленым
   - Иконка "▶️"
   - Кнопка "Начать" активна

2. **is_ready = false**:
   - Затемненный фон
   - Кнопки действий disabled
   - Подсказка "Ожидает выполнения задачи: ..."

## 5.4 Зависимости - UX паттерны

### ✅ ПРИ СОЗДАНИИ СВЯЗИ:

```javascript
// 1. Пользователь выбирает задачу
// 2. Перетаскивает ребро к другой задаче
// 3. ОТПРАВЛЯЕМ ОДИН ЗАПРОС
POST /projects/{slug}/tasks/{id}/dependencies
{
  "target_task_id": 456,
  "dependency_type": "blocks"
}

// 4. Optimistic update - добавляем ребро сразу
// 5. При ошибке - откатываем
```

### 🚫 ЗАПРЕЩЕНО:
- Создавать циклические зависимости (бэкенд вернет 400)
- Создавать дубликаты (бэкенд вернет 400)
- Создавать зависимость от выполненной задачи (логически ок, но бесполезно)

## 5.5 Зависимости и ребра

В ответе `GET .../tasks/graph` ребро отражает запись зависимости: тип связи (`type`), опционально `description` в `data`, подпись/анимация для UX. **Отдельного публичного API для настройки «действий на ребре» в стиле нотификаций через REST в текущей версии нет** — не закладывайте экраны управления такими действиями, пока бэкенд их не экспонирует.

# ⚠️ ОБРАБОТКА ОШИБОК

## 8.1 Коды статусов и их значение

| Код | Значение | Действие фронта |
|-----|----------|-----------------|
| **400** | Bad Request | Показать ошибку валидации |
| **401** | Unauthorized | Redirect на /login |
| **403** | Forbidden | Скрыть UI, показать "Нет доступа" |
| **404** | Not Found | 404 страница |
| **409** | Conflict | Предложить обновить данные |
| **422** | Validation Error | Подсветить поля |
| **500** | Server Error | Retry с экспоненциальной задержкой |

## 8.2 UX стратегии при ошибках

### ✅ ДЛЯ CRUD:

```javascript
// CREATE - показать ошибку, сохранить данные в localStorage
// READ - показать скелетон, retry
// UPDATE - откатить оптимистичное обновление
// DELETE - подтверждение повторно
```

### 🚨 КРИТИЧЕСКИЕ ОШИБКИ:

1. **401 при запросе с токеном**:
   - Попытаться refresh
   - Если не удалось - logout + redirect

2. **403 при попытке действия**:
   - НЕ ПОКАЗЫВАТЬ кнопку вообще!
   - Если показали - скрыть после ошибки

3. **500 при загрузке графа**:
   - Retry 3 раза
   - Предложить обновить страницу

# 🔐 1. АУТЕНТИФИКАЦИЯ И ПОЛЬЗОВАТЕЛИ

## 1.1 Регистрация нового пользователя

**POST** `/api/v1/auth/register`

**Назначение:** Создание учётной записи. Пользователь **не** получает токены сразу — на указанный **email** отправляется 6-значный код.

**Параметры запроса:**
| Поле | Тип | Обязат. | Описание | Валидация |
|------|-----|---------|----------|-----------|
| `first_name` | string | ✅ | Имя | 1-100 символов |
| `last_name` | string | ✅ | Фамилия | 1-100 символов |
| `username` | string | ✅ | Логин | 3-50 символов, только a-z0-9_.- |
| `password` | string | ✅ | Пароль | ≥8 символов, заглавные+строчные+цифры |
| `email` | string | ✅ | Email | Валидный формат, обязателен |

**Успешный ответ (200):**
```json
{
  "requires_verification": true,
  "user_id": 123,
  "message": "Verification code sent to your email",
  "verification_code": null,
  "email_sent": true
}
```

Поле `verification_code` может быть заполнено только при отладке или если письмо не отправлено (`email_sent: false`).

**Ошибки:** 400 — занят username/email; 422 — невалидное тело.

---

## 1.2 Вход в систему

**POST** `/api/v1/auth/login`

**Назначение:** Аутентификация. Если `email_verified` у пользователя ещё `false`, возвращается запрос на ввод кода с почты (новый код отправляется на email).

**Успешный ответ — email подтверждён:** `requires_verification: false`, `access_token`, `refresh_token`, `user`, …

**Успешный ответ — нужен код:**
```json
{
  "requires_verification": true,
  "user_id": 123,
  "verification_code": null,
  "email_sent": true
}
```

**Ошибки:** 401 — неверные учётные данные.

---

## 1.3 Подтверждение email (код из письма)

**POST** `/api/v1/auth/verify-email`

**Параметры запроса:**
| Поле | Тип | Обязат. | Описание |
|------|-----|---------|----------|
| `user_id` | int | ✅ | ID пользователя |
| `code` | string | ✅ | 6 символов из письма |

**Успешный ответ (200):** как у обычного логина — `access_token`, `refresh_token`, `user`, …

**Ошибки:** 400 — неверный или просроченный код, лимит попыток.

---

## 1.4 Обновление токена

**POST** `/api/v1/auth/refresh`

**Назначение:** Получение новой пары токенов по refresh token. Делать **ЗА 5 МИНУТ** до истечения access token.

**Параметры запроса:**
| Поле | Тип | Обязат. | Описание |
|------|-----|---------|----------|
| `refresh_token` | string | ✅ | Refresh token из предыдущей сессии |

**Успешный ответ (200):**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "bearer",
  "expires_at": "2026-02-13T16:30:00Z",
  "user": {
    "id": 123,
    ...
  }
}
```

**Ошибки:**
| Код | Причина | Действие фронта |
|-----|--------|-----------------|
| 401 | Невалидный/просроченный refresh token | Logout, redirect на /login |

**Оптимизация:**
- Хранить `expires_at` и запускать таймер за 5 минут до истечения
- Не ждать 401 ответа

## 1.5 Выход из системы

**POST** `/api/v1/auth/logout`

**Назначение:** Завершение текущей сессии. Токен добавляется в черный список.

**Параметры:** Нет (токен в заголовке Authorization)

**Успешный ответ (200):**
```json

}
```

**Ошибки:** Нет (даже с невалидным токеном вернет 200)

**Оптимизация:** Очистить localStorage/sessionStorage, редирект на /login.

## 5.6 Добавление действия на зависимость

**POST** `/api/v1/projects/{project_slug}/tasks/dependencies/{dependency_id}/actions`

**Назначение:** Добавление бизнес-логики на ребро графа.

**Параметры запроса:**
| Поле | Тип | Обязат. | Для каких типов |
|------|-----|---------|-----------------|
| `action_type_code` | string | ✅ | Все |
| `target_user_username` | string | ❌* | `notify_custom`, `create_subtask` |
| `target_status` | string | ❌* | `change_status` |
| `message_template` | string | ❌* | `notify_*` |
| `delay_minutes` | int | ❌ | Все (0 = немедленно) |
| `execute_order` | int | ❌ | Все |

*Обязательно для указанных типов

**Типы действий:**
| Код | Описание | Обязательные поля |
|-----|----------|-------------------|
| `notify_assignee` | Уведомить исполнителя целевой задачи | `message_template` |
| `notify_creator` | Уведомить создателя исходной задачи | `message_template` |
| `notify_custom` | Уведомить конкретного пользователя | `target_user_username`, `message_template` |
| `change_status` | Изменить статус целевой задачи | `target_status` |
| `create_subtask` | Создать подзадачу | `target_user_username` |

**Успешный ответ (200):** `DependencyActionResponse`

**Ошибки:**
| Код | Причина |
|-----|--------|
| 400 | Отсутствуют обязательные поля |
| 403 | Нет прав (только owner/manager) |
| 404 | Зависимость не найдена |
